# フェイズA（保有銘柄ブリーフ）要件定義書（現行運用版）

最終更新: 2026-03-01（JST）

本ドキュメントは、保有銘柄の「売り時/買い増し判断支援」を対象としたフェイズAの正本要件です。  
フェイズB（新規銘柄リサーチ）は本仕様の対象外とし、別途定義します。

## 1. 背景と目的

1. 中長期・本業併行の運用前提で、日々の雰囲気判断を減らす。
2. BOTは売買を断言せず、例外検知と行動候補の提示に徹する。
3. 「まず損失回避・集中リスク制御」を最優先とする。

## 2. 対象範囲（In Scope）

1. 保有銘柄の毎日ブリーフ通知（Discord）
2. Core/Satellite/Sandboxの役割分離（watchlist priorityを利用）
3. Risk Level（RL）とEarnings Gate（E）の判定
4. 通知ログ記録とクールダウン制御
5. 日次ジョブ連携（`scripts/run_daily_job.py --enable-phase-a`）

## 3. 対象外（Out of Scope）

1. フェイズB（新規銘柄スクリーニング/深掘り）
2. 銘柄の自動売買執行
3. 月次の銘柄提案自動生成
4. Obsidian自動書き込み

## 4. 現在実装済み要件（2026-03-01時点）

1. フェイズAは `--enable-phase-a` で明示有効化できる。
2. 既存日次ジョブと同じ通知チャネル/通知タイミング制御に従う。
3. 判定項目は以下を利用する。
   - 日次騰落率
   - 週次騰落率
   - 移動平均（短期/長期）
   - 決算まで日数
4. RLを `RL0-3`、決算ゲートを `E0-3`（決算不明は `E1(不明)`）で出力する。
5. Coreは常時候補、Satellite/Sandboxは例外時のみ候補に入れる。
6. 1件の通知失敗で全体停止しない（銘柄単位で継続）。
7. 日次ジョブJSONサマリにフェイズA結果を合算する。
8. 同一ジョブ内の市場データ取得は銘柄単位キャッシュを使う。

## 5. 未実装要件（残タスク）

1. 保有額/総資産を使った可変Risk Budget（Core比率上限の動的制御）
2. RL判定への出来高異常・ギャップ・IR異常の組み込み
3. フェイズAしきい値の`/ops`管理（global_settings/runtime）
4. フェイズA専用の運用ダッシュボード表示（履歴・根拠・実行結果）
5. フェイズA通知文面の最終フォーマット確定（運用チューニング）
6. 監視/保有ステータスの厳密分離（保有銘柄のみ対象実行オプション）

## 6. 受け入れ基準（全体）

1. サイレント失敗を起こさない（処理失敗はログと結果件数に反映）。
2. JST基準で日次実行し、既存ジョブ挙動を壊さない（オプトイン維持）。
3. 通知はクールダウン制御下で重複抑制される。
4. 既存テスト群と追加テストがすべて成功する。
5. ドキュメント、実装、運用コマンドが一致している。

## 7. 実行フェーズ計画（フェイズB除外）

### Phase A-1: データ基盤整備（最優先）

目的: 可変Risk Budgetの前提データを整える。

タスク:

1. 保有情報スキーマ追加（保有株数、平均取得単価、評価額、総資産）
2. Firestore repository/API更新
3. `/ops` または watchlist更新APIで保有情報を編集可能にする
4. 既存データへの移行方針を定義（欠損時は判定保留）

受け入れ基準:

1. 保有情報が保存/取得/更新できる
2. 欠損時は `【データ不明】` か `判定保留` を明示
3. テスト追加（repository/api/service）

### Phase A-2: 判定ロジック拡張

目的: RL/E判定を実運用レベルに引き上げる。

タスク:

1. 可変Risk Budget実装（例: RL0=70 / RL1=60 / RL2=50 / RL3=40）
2. `現Core比率` と `超過量`（%/金額）の算出
3. 決算ゲート段階制限（E3禁止/E2 1/3制限/E1分割推奨）
4. 追加リスク要因（出来高、ギャップ、トレンド崩れ）をRLに統合

受け入れ基準:

1. 通知に「許容上限/現状比率/超過有無」が出る
2. 縮小候補の提示根拠が定量で追跡可能
3. 単体テストで閾値境界を網羅

### Phase A-3: 設定の運用化

目的: コード変更なしで運用チューニング可能にする。

タスク:

1. `global_settings/runtime` にフェイズA設定を追加
2. `/admin/settings` API・Web `/ops` から編集可能にする
3. 既定値と優先順位を文書化（runtime > env > default）

受け入れ基準:

1. しきい値変更が次回ジョブから反映される
2. 不正設定はバリデーションで拒否される
3. 変更履歴（updated_at/updated_by）が追跡できる

### Phase A-4: 運用可視化と検証

目的: 実運用の改善ループを回せる状態にする。

タスク:

1. 通知ログのフェイズA絞り込みビュー追加
2. 日次の「要対応件数」「RL分布」「通知スキップ理由」を集計
3. 週次レビュー手順（見送り/縮小/維持の結果確認）を運用手順へ反映

受け入れ基準:

1. 1週間分の判定推移をUIまたはログで追える
2. スキップ理由の分析が可能
3. 運用手順書と実装が一致
