# トレーダー潜在要件ベース 改善アイデア一覧

最終更新: 2026-02-19（JST）

## 1. 目的

- 実装完了後の磨き込みに向けて、トレーダーが文面にしづらい潜在要件を仮説化し、改善案を整理する。
- 本書は仕様の正本ではなく、Issue化前の検討バックログとして扱う。

## 2. 潜在要件の仮説（共通）

1. 通知が来た瞬間に「何をすべきか」を判断したい。
2. 有益な通知だけ受け取り、ノイズと見落としを同時に減らしたい。
3. 数値だけでなく「今が異常なのか平常なのか」を知りたい。
4. 売買判断の検討履歴を、後で検証できる形で残したい。
5. 監視運用の手間を増やさず、判断品質だけ上げたい。

## 3. 改善アイデア（短期: 1〜2スプリント）

### A-01 通知文の結論先出し化

- 仮説: 忙しい時間帯は長文を読まずに見逃す。
- 改善案: 通知1行目を `優先度/推奨アクション/根拠数値` の固定フォーマットに統一する。
- 期待効果: 通知確認から行動判断までの時間短縮。
- MVP整合: あり（通知フォーマット改善）。

### A-02 「なぜ今通知されたか」の差分明示

- 仮説: 現在値だけでは納得感が弱い。
- 改善案: `現在値 - 各中央値` と `乖離率` を通知本文に追加する。
- 期待効果: 判断理由の透明化、誤認識の低減。
- MVP整合: あり。

### A-03 新規シグナルと継続シグナルの区別強化

- 仮説: 継続通知と新規発生通知は価値が異なる。
- 改善案: 通知カテゴリに `新規` / `継続` / `解除` を明示する。
- 期待効果: 優先順位づけが容易になる。
- MVP整合: あり（状態遷移表示の強化）。

### A-04 データ信頼度の可視化

- 仮説: 取得成功でもソース品質差が判断に影響する。
- 改善案: 使用データソース（株探/Yahoo/J-Quants）と取得時刻を通知ログに保持・表示する。
- 期待効果: データ起因の誤判断リスク低減。
- MVP整合: あり。

### A-05 データ不明通知の再発防止導線

- 仮説: `【データ不明】` を見ても次アクションが分かりにくい。
- 改善案: データ不明通知に「次の確認先」（例: 履歴、運用操作）を固定文で付与する。
- 期待効果: 障害対応初動の短縮。
- MVP整合: あり。

### A-06 ウォッチリストの優先度タグ

- 仮説: 100銘柄を同じ重みで監視すると運用が重い。
- 改善案: `高/中/低` 優先度タグを追加し、一覧と通知ログで絞り込み可能にする。
- 期待効果: 重要銘柄の見落とし防止。
- MVP整合: あり（既存管理機能の拡張）。

### A-07 決算接近フラグの前倒し表示

- 仮説: 割安通知でも決算直前は解釈が変わる。
- 改善案: `決算までN日` を一覧と通知に表示する。
- 期待効果: イベントリスクを踏まえた判断がしやすくなる。
- MVP整合: あり（既存決算データ活用）。

### A-08 朝イチ要約（前日からの変化のみ）

- 仮説: 日中通知だけでは全体像を俯瞰しづらい。
- 改善案: 毎営業日朝に `新規シグナル/解除/データ不明` のサマリ通知を1通送る。
- 期待効果: 監視開始時の状況把握が早くなる。
- MVP整合: 条件付き（運用ジョブ追加は必要）。

### A-09 通知スキップ理由のUI露出

- 仮説: 「通知が来ない理由」が分からないと不安が残る。
- 改善案: クールダウン・条件未達・OFF設定などのスキップ理由を銘柄単位で可視化する。
- 期待効果: 問い合わせ削減、運用信頼性向上。
- MVP整合: あり。

### A-10 履歴の「判断メモ」標準化

- 仮説: 理由メモが自由記述だけだと後で分析しにくい。
- 改善案: 理由メモにテンプレート（仮説/トリガー/見直し日）を導入する。
- 期待効果: 振り返り品質向上、チーム共有の標準化。
- MVP整合: あり。

## 4. 改善アイデア（中期: 3〜5スプリント）

### B-01 銘柄ごとの通知強度プロファイル

- 仮説: 銘柄特性によって適切な通知頻度が異なる。
- 改善案: 銘柄単位でクールダウン時間や通知条件の厳しさを設定可能にする。
- 期待効果: ノイズ削減と機会損失抑制の両立。
- MVP整合: 一部拡張（ルール追加）。

### B-02 Regime（相場環境）メタ情報の付与

- 仮説: 同じPER/PSR水準でも相場局面で意味が変わる。
- 改善案: TOPIX等の簡易地合い指標を通知に併記し、平常/警戒を示す。
- 期待効果: 文脈なき逆張り判断の抑制。
- MVP整合: 拡張領域。

### B-03 通知有効性スコア

- 仮説: 「どの通知が役に立ったか」を測れないと改善が止まる。
- 改善案: Discordリアクション等を軽量フィードバックとして収集し、有効性を可視化する。
- 期待効果: 通知品質の継続改善サイクルを構築。
- MVP整合: 拡張領域。

### B-04 監視戦略テンプレート

- 仮説: 銘柄登録時の設定判断が毎回重い。
- 改善案: `順張り監視` `逆張り監視` などの設定プリセットを提供する。
- 期待効果: 設定ミス削減、導入時間短縮。
- MVP整合: 拡張領域。

### B-05 週次の振り返りレポート自動生成

- 仮説: 日次運用だけでは改善ポイントが見えにくい。
- 改善案: 週次で `通知件数/データ不明率/未確認通知` を集計し配信する。
- 期待効果: 運用品質の可視化、改善施策の優先順位づけ。
- MVP整合: 拡張領域。

## 5. Issue化テンプレート（本書から転記）

1. 概要: 対象アイデアID（例: A-03）
2. 背景/目的: 潜在要件仮説のどれを解くか
3. スコープ: 画面/API/通知文面/運用のどこを変えるか
4. 受け入れ基準:
   - Given: 前提データ・設定
   - When: 操作またはジョブ実行
   - Then: 可視化・通知・保存結果
5. 非スコープ: 今回やらないこと
6. ロールバック方針: 既存運用へ戻す条件

## 6. 優先着手候補（提案）

1. A-01 通知文の結論先出し化
2. A-03 新規/継続/解除の区別強化
3. A-09 通知スキップ理由のUI露出
4. A-06 ウォッチリスト優先度タグ
5. A-07 決算接近フラグ表示

上記5件は、既存MVPの範囲内でトレーダー体験を直接改善しやすく、運用リスクを増やしにくい。
