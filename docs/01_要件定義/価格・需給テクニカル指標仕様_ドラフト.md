# 価格・需給テクニカル指標仕様（現行実装）

最終更新: 2026-03-08（JST）

このドキュメントは、`J-Quants LITE` を前提とした価格・需給テクニカル指標の現行仕様です。  
ファイル名は互換のため `ドラフト` を残しているが、内容は実装済みの正本として扱う。  
現行運用で参照する定義を固定し、将来の拡張時も解釈が割れないようにすることを目的とします。

## 1. 目的

- `J-Quants LITE` の日足データから算出する価格・需給系テクニカル指標の定義を整理する。
- 実装時にブレやすい `基準価格` `期間` `例外処理` `フラグ判定` を先に固定する。
- 今回確定した項目を正本候補として整理し、未整理の項目は今後同じ形式で追記できるようにする。

## 2. スコープ

- 対象は `価格系 / トレンド系 / 高値安値系 / 需給系 / 日足強弱系 / 流動性・安定性指標` のうち、`J-Quants LITE` の現行データから直接取得または自前計算できる項目とする。
- `A〜I` の全項目を本書末尾に一覧化する。
- 現時点の `J-Quants V2 / LITE` で取得前提を置けない項目は、本書のスコープから除外する。

## 3. 前提

- データソースは `J-Quants LITE` を前提とする。
- トレンド、高値安値、騰落率、ボラティリティは `調整後価格` ベースで計算する。
- `ATR` は補助項目 `true_range` の定義に従って計算する。
- ローソク足、ヒゲ、ギャップは `当日OHLC` ベースで計算する。
- 日数は `営業日ベース` で扱う。
- 履歴不足時は、数値項目は `null`、フラグ項目は `false` を返す。
- `new_high_* / new_low_*` は `当日を除く過去N営業日` と比較する。
- `52w_*` の内部定義は `直近252営業日` とする。
- `ytd_*` は `その年の1月最初の営業日から当日まで` とする。

## 4. 用語

- `adj_close`: 調整後終値
- `adj_open / adj_high / adj_low / adj_close`: 調整後OHLC
- `ma_n`: 直近 `n` 営業日の調整後終値単純移動平均
- `prev_close`: 前営業日の当日OHLC基準の終値
- `high / low / open / close`: 当日OHLC基準の値

## 5. 廃止・置換

| 旧項目名 | 対応 | 日本語名 | 説明 |
|---|---|---|---|
| `days_from_recent_high` | 廃止 | 直近高値からの経過日数 | 定義が曖昧なため廃止する |
| - | `days_from_20d_high` に置換 | 20日高値からの経過日数 | 短期高値更新後の経過営業日数を見る用途 |
| - | `days_from_ytd_high` に置換 | 年初来高値からの経過日数 | 年初来高値からの経過営業日数を見る用途 |

## 6. 追加・変更項目

| 項目名 | 日本語名 | 型 | 計算式 / 判定式 | 例外時処理 | 説明 |
|---|---|---|---|---|---|
| `days_from_20d_high` | 20日高値からの経過日数 | `integer \| null` | 当日を含む直近20営業日内の最高値を最後に付けた日からの経過営業日数 | 履歴不足時は `null` | 短期高値更新後、何営業日経過したかを表す |
| `days_from_ytd_high` | 年初来高値からの経過日数 | `integer \| null` | 年初来高値を最後に付けた日からの経過営業日数 | 年初来データ不足時は `null` | 年初来高値からの調整期間を表す。同値高値が複数ある場合は最後に付けた日を採用する |
| `body_ratio` | 実体比率 | `float \| null` | `abs(close - open) / (high - low)` | `high == low` のとき `0` | 当日の値幅に対して実体が占める割合 |
| `true_range` | 真の値幅 | `float \| null` | `max(high - low, abs(high - prev_close), abs(low - prev_close))` | `prev_close` 欠損時は `null` | ATR算出の元になる当日の実質値幅 |
| `above_ma5` | 5日線上フラグ | `bool` | `adj_close > ma_5` | `ma_5 == null` のとき `false` | 調整後終値が5日移動平均線を上回っているか |
| `cross_up_ma5` | 5日線上抜け | `bool` | `adj_close_t-1 <= ma_5_t-1` かつ `adj_close_t > ma_5_t` | 前日値またはMA不足時は `false` | 前日は5日線以下、当日は5日線超えになったか |
| `cross_down_ma5` | 5日線下抜け | `bool` | `adj_close_t-1 >= ma_5_t-1` かつ `adj_close_t < ma_5_t` | 前日値またはMA不足時は `false` | 前日は5日線以上、当日は5日線未満になったか |

## 7. 補助項目

| 項目名 | 日本語名 | 型 | 計算式 / 判定式 | 例外時処理 | 説明 |
|---|---|---|---|---|---|
| `close_position_in_range` | 当日値幅内終値位置 | `float \| null` | `(close - low) / (high - low)` | `high == low` のとき `0.5` | 当日の値幅の中で終値がどこにあるかを表す |
| `upper_shadow_ratio` | 上ヒゲ比率 | `float \| null` | `(high - max(open, close)) / (high - low)` | `high == low` のとき `0` | 当日の値幅に対する上ヒゲの割合 |
| `lower_shadow_ratio` | 下ヒゲ比率 | `float \| null` | `(min(open, close) - low) / (high - low)` | `high == low` のとき `0` | 当日の値幅に対する下ヒゲの割合 |
| `gap_up_down_pct` | ギャップ率 | `float \| null` | `(open / prev_close - 1) * 100` | `prev_close` 欠損時は `null` | 始値が前日終値から何％離れて始まったかを表す |

## 8. candle_type

`candle_type` は以下の固定分類を採用する。

### 8.1 判定優先順位

`flat_bar -> upper_rejection -> lower_rejection -> doji -> bull -> bear`

短期売買用途では、実体の小ささよりヒゲの意味を優先して拾うため、この順序で判定する。

### 8.2 分類定義

| 値 | 日本語名 | 判定式 | 説明 |
|---|---|---|---|
| `flat_bar` | 値幅なし足 | `high == low` | 値幅がない足 |
| `upper_rejection` | 上ヒゲ優勢足 | `upper_shadow_ratio >= 0.4` かつ `body_ratio <= 0.35` | 上ヒゲが長く、上値を抑えられた足 |
| `lower_rejection` | 下ヒゲ優勢足 | `lower_shadow_ratio >= 0.4` かつ `body_ratio <= 0.35` | 下ヒゲが長く、下値を拾われた足 |
| `doji` | 十字線系 | `high != low` かつ `abs(close-open)/(high-low) <= 0.1` | 実体が小さい足 |
| `bull` | 陽線 | `close > open` かつ上記いずれにも非該当 | 買い優勢で引けた足 |
| `bear` | 陰線 | `close < open` かつ上記いずれにも非該当 | 売り優勢で引けた足 |

## 9. 例外処理

| 条件 | 項目名 | 処理 |
|---|---|---|
| `high == low` | `close_position_in_range` | `0.5` を返す |
| `high == low` | `body_ratio` | `0` を返す |
| `high == low` | `upper_shadow_ratio` | `0` を返す |
| `high == low` | `lower_shadow_ratio` | `0` を返す |
| `high == low` | `candle_type` | `flat_bar` を返す |
| 履歴不足 | 数値項目 | `null` を返す |
| 履歴不足 | フラグ項目 | `false` を返す |

| `high_52w_near` | 52週高値接近フラグ | `bool` | `adj_close >= high_52w * 0.97` | `high_52w == null` のとき `false` | 調整後終値が52週高値の3%以内にあるか |

## 12. 流動性・安定性指標

| `atr_pct_14` | 14日ATR率 | `float \| null` | `(atr_14 / adj_close) * 100` | `atr_14 == null` または `adj_close == null` のとき `null` | ATRを株価に対する割合に換算した指標 |

## 14. 保存・差分更新設計（ドラフト）

本章は、`200日MA` や `52週高値` のような長期窓を毎日APIから全取得せず、ローカル保存済みデータを使って差分更新するための設計方針である。  
現行運用の `daily_metrics` は `PER / PSR` 用の正本であり、価格・需給テクニカル指標は別コレクションで扱う前提とする。

### 14.1 設計方針

1. `J-Quants LITE` からは毎日 `差分 + 重複取得分` だけ取得する。
2. 日次の算出は、APIレスポンスをそのまま使い回さず、DBに保存した `日足バー` を正本として行う。
3. `raw input` と `derived indicator` は別コレクションに分ける。
4. `daily_metrics` には混在させず、既存のPER/PSR運用を壊さない。
5. 日次ジョブは `active watchlist` 銘柄を対象にし、まずは全市場一括ではなく監視対象だけを回す。

### 14.2 追加コレクション案

| コレクション名 | doc id | 役割 | 主なフィールド |
|---|---|---|---|
| `price_bars_daily` | `{ticker}\|{trade_date}` | J-Quantsから取得した日足バーの保存先 | `ticker`, `trade_date`, `open`, `high`, `low`, `close`, `volume`, `turnover_value`, `adj_open`, `adj_high`, `adj_low`, `adj_close`, `adj_volume`, `source`, `fetched_at` |
| `technical_indicators_daily` | `{ticker}\|{trade_date}` | 日次算出した価格・需給テクニカル指標の保存先 | `ticker`, `trade_date`, `schema_version`, `calculated_at`, `close_vs_ma25`, `drawdown_from_52w_high`, `turnover_ratio`, `volatility_20d` など |
| `technical_sync_state` | `{ticker}` | 差分取得と再計算の進捗管理 | `ticker`, `latest_fetched_trade_date`, `latest_calculated_trade_date`, `last_fetch_from`, `last_fetch_to`, `last_run_at`, `last_status`, `last_error` |
| `technical_alert_rules` | `{ticker}\|{rule_id}` | 銘柄ごとの技術アラート条件 | `rule_name`, `field_key`, `operator`, `threshold_value`, `threshold_upper`, `is_active`, `note`, `updated_at` |
| `technical_alert_state` | `{ticker}\|{rule_id}` | 前回判定結果と前回発火状態 | `last_evaluated_trade_date`, `last_condition_met`, `last_triggered_at`, `updated_at` |

### 14.3 日次更新フロー

1. `watchlist` から `is_active=true` の銘柄を取得する。
2. 各銘柄の `technical_sync_state` を読む。
3. 初回同期:
   - `J-Quants LITE` で取得可能な範囲を一括取得し、`price_bars_daily` に保存する。
4. 差分同期:
   - `latest_fetched_trade_date` がある場合、`latest_fetched_trade_date - 30暦日` から当日までを再取得する。
   - 直近を重複取得するのは、訂正、反映遅延、分割・併合に伴う調整値更新に備えるためである。
5. 取得したバーを `price_bars_daily` に `upsert` する。
6. `price_bars_daily` から対象銘柄の必要履歴を読み込む。
7. `technical_indicators_daily` を再計算し、対象期間を `upsert` する。
8. 成功時のみ `technical_sync_state` の `latest_fetched_trade_date` と `latest_calculated_trade_date` を更新する。
9. 保存済み `technical_alert_rules` を評価し、発火時は Discord 通知と `notification_log` 保存を行う。
10. `technical_alert_state` に前回判定結果と前回発火日時を残す。
11. ジョブ全体の成否は `job_run` に残す。

### 14.4 再計算ウィンドウ

| 項目 | 推奨値 | 意図 |
|---|---|---|
| API再取得オーバーラップ | `30暦日` | 直近修正、祝日跨ぎ、調整値更新の吸収 |
| ローカル読込範囲 | `直近520営業日` または保存済み全件 | `200日MA`, `52週高値`, `slope_ma200` を含む長期窓の再計算に必要 |
| 再書込範囲 | `直近260営業日` | `52週系` と `200日系` の最新値を安全に再計算するため |

- `52週高値/安値` は `252営業日` を使うため、再計算対象日の前段履歴も含める必要がある。
- `slope_ma200` は `ma200_t` と `ma200_t-20` を使うため、単純な `200本` より余裕を持たせる。
- `J-Quants LITE` の保持上限が約2年である間は、実質的に `520営業日前後` を読めば足りる想定とする。

### 14.5 保持期間

1. `price_bars_daily` は、`J-Quants LITE` で取得できる期間をすべて保持する。
2. `technical_indicators_daily` も、原則として同期間を保持する。
3. `technical_sync_state` は銘柄ごとに1件だけ保持する。
4. 現行MVPの監視対象上限 `100銘柄` では、Firestoreの件数規模として許容範囲に収まる前提とする。

### 14.6 定期補正

1. 日次ジョブは差分取得を基本とする。
2. 補正用に、`週1回` もしくは `手動実行` で `保有期間全件` の再取得ジョブを用意する。
3. 株式分割・併合などで調整後価格が広範囲に更新された場合は、この全件再同期で吸収する。

### 14.7 失敗時ポリシー

1. `price_bars_daily` の保存に失敗した銘柄は、その銘柄の `technical_indicators_daily` を更新しない。
2. 指標計算に失敗した場合でも、取得済み `price_bars_daily` は保持する。
3. `technical_sync_state` は成功した段階までしか進めない。
4. 欠損や計算不能はサイレントに落とさず、`job_run` とアプリログで追跡可能にする。
5. `technical_indicators_daily` には `schema_version` を持たせ、計算式変更時の再計算対象を識別できるようにする。

## 15. 実装メモ

- `cross_up_ma5 / cross_down_ma5` は、前日は前日の `ma_5`、当日は当日の `ma_5` と比較する。
- `cross_up_ma200 / cross_down_ma200` も同様に、前日は前日の `ma_200`、当日は当日の `ma_200` と比較する。
- `true_range` は厳密性を優先し、`prev_close` 欠損時は `null` とする。
- `drawdown_from_52w_high` は負値で表現する。必要に応じて表示層で絶対値表記へ変換してよい。
- `時価総額` は一般に `close * shares_outstanding` で計算できるが、現時点の `J-Quants LITE` 公式カラムだけでは `shares_outstanding` の日次取得前提を置けないため、本書のスコープから除外する。
- `配当落ちイベント補正` は、現時点の `J-Quants V2 / LITE` で直接取得前提を置けないため、本書のスコープから除外する。
- 本書は今回合意した範囲の固定版であり、今後の仕様追加も同じ表形式で追記する。
