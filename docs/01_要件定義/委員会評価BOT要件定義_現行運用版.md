# 委員会評価BOT要件定義書（現行運用版）

最終更新: 2026-03-01（JST）

本ドキュメントは、銘柄そのものの評価にフォーカスした「委員会評価BOT」の正本要件です。  
ポジションサイズ（いくら持つか）はユーザー判断とし、BOTは断言しません。  
フェイズB（新規銘柄リサーチ）は本仕様の対象外とし、別途定義します。

## 1. 背景と目的

1. 中長期・本業併行の運用前提で、日々の雰囲気判断を減らす。
2. BOTは売買を断言せず、銘柄状態の説明責任に徹する。
3. 観点の偏りを避けるため、複数観点の委員会方式を採用する。

## 2. 対象範囲（In Scope）

1. 対象は `watchlist` のうち `評価対象チェックあり` 銘柄
2. 6観点委員会評価（事業/財務/バリュ/テクニカル/イベント/リスク）
3. 2軸ラベル出力（自信 × 強さ）
4. Discord通知（粒度はwatchlist設定で制御）
5. 通知ログ記録とクールダウン制御
6. 日次ジョブ連携（評価ジョブ）

## 3. 対象外（Out of Scope）

1. フェイズB（新規銘柄スクリーニング/深掘り）
2. 銘柄の自動売買執行
3. ポジションサイズ最適化（保有比率の自動提案）
4. Obsidian自動書き込み

## 4. 現在実装済み要件（2026-03-01時点）

1. watchlistに `evaluation_enabled` と通知粒度設定（`ALL/TOP_N/ALERT_ONLY`）を実装済み。
2. 委員会6観点評価エンジン（事業/財務/バリュ/テクニカル/イベント/リスク）を実装済み。
3. 2軸ラベル（自信×強さ、各1-5）を通知/ログ/APIで扱える状態に実装済み。
4. 旧フェイズA（保有ブリーフ）を日次ジョブから置換済み（`run_daily_job.py` は委員会評価を既定実行）。
5. `/notifications/logs` に委員会向け絞り込み（カテゴリ・自信下限・強さ下限）と7日サマリAPIを実装済み。
6. 月次基礎調査の3層保存（`raw/structured/summary`）と失敗時の前回データ継続利用を実装済み。
7. `/ops` 全体設定に「委員会日次時刻」「基礎調査月次時刻」を追加済み。
8. 基礎調査に EDINET/e-Stat の任意エンリッチ経路を追加済み（APIキー設定時のみ実行、未設定時は既存経路で継続）。
9. e-Stat のAPIエラー/最新値欠損時は `enrichment_errors` と `summary.macro_note` で欠損理由を可視化する実装を追加済み。

## 5. 未実装要件（残タスク）

1. 四季報/IR/短信/有報の収集経路を本番データ源で拡充（EDINET/e-Statは初期連携済み、四季報・TDnet・IR本文解析は拡張継続）。
2. 四季報の取得制約に備えた手動投入経路（`source=manual`）の運用UI整備。
3. 月次基礎調査ジョブのCloud Scheduler反映運用（管理画面設定値との完全同期）を運用手順へ統合。
4. 基礎調査失敗一覧の管理画面内可視化（現在は通知/ログベース）。

## 6. 受け入れ基準（全体）

1. サイレント失敗を起こさない（処理失敗はログと結果件数に反映）。
2. JST基準で日次実行し、既存運用機能を壊さない。
3. 通知はクールダウン制御下で重複抑制される。
4. 評価は売買断言をしない。
5. 既存テスト群と追加テストがすべて成功する。
6. ドキュメント、実装、運用コマンドが一致している。

## 7. 委員会観点定義（初期）

1. 事業（Qual）: 競争優位、成長ドライバー、事業の崩れ兆候
2. 財務（BS/CF）: 財務健全性、CF、安全余力
3. バリュ（Valuation）: 過去レンジと同業比較での位置
4. テクニカル（Price/Volume）: トレンド、ボラ、出来高異常
5. イベント（Macro/Event）: 決算・IR・制度変更など直近イベント
6. リスク（Risk）: 見落としやすい悪材料、データ欠損、判断保留条件
7. 各観点コメントは「最大3行」とし、冗長な長文を禁止する。

## 8. 2軸ラベル定義（初期）

1. 自信（Confidence）: `1-5`
   - `1`: 根拠不足・欠損多い
   - `3`: 根拠はあるが対立観点あり
   - `5`: 根拠十分・整合性が高い
2. 強さ（Strength）: `1-5`
   - `1`: 影響が小さい
   - `3`: 注意が必要
   - `5`: 影響が大きい
3. 方向（ポジ/中立/ネガ）は2軸に含めず、観点コメントとして別途出力する。

## 9. データ要件（初期MVP + 追加推奨）

初期MVP（必須）:

1. 終値、日次/週次騰落率
2. 出来高（20日平均比）
3. 決算日（次回）
4. PER/PSR（取得可能範囲）

保存粒度（必須）:

1. `raw`: 元データ本文/取得メタデータ
2. `structured`: 指標化・正規化済みデータ
3. `summary`: 委員会入力向け要約データ

追加推奨（精度向上）:

1. PBR、配当利回り
2. 売買代金（流動性）
3. セクター情報（同業比較）
4. 主要財務指標（営業CF、FCF、自己資本比率）
5. ガイダンス修正/重要IRフラグ

情報源ポリシー:

1. 信頼性は「情報源の固定順位」ではなく、ドキュメント単位で評価する。
2. 信頼度算定は最低限以下を用いる。
   - 一次情報性（企業公式・法定開示か）
   - 更新日時（鮮度）
   - 根拠の明示性（数値・出典URLの有無）
3. 四季報/IR/短信/有報/その他は候補ソースとして保持し、最終評価は信頼度スコアで決定する。

## 10. 通知仕様（watchlist設定化）

1. `evaluation_enabled`（bool）: 評価対象か（既定値: `false`）
2. `evaluation_notify_mode`（enum）:
   - `ALL`: 対象銘柄を全件通知
   - `TOP_N`: 強さ上位N件のみ通知
   - `ALERT_ONLY`: 強さが閾値以上の銘柄のみ通知
   - 既定値: `ALERT_ONLY`
3. `evaluation_top_n`（int, mode=TOP_Nで使用）
4. `evaluation_min_strength`（int, mode=ALERT_ONLYで使用）
5. `evaluation_scheduled_time`（HH:MM, JST）: 評価ジョブ実行時刻（管理ページで変更可能、既定値 `18:00`）
6. 月次基礎調査ジョブ実行時刻（管理ページで変更可能、既定値: 毎月1日 `18:00` JST）

## 11. 実行フェーズ計画（フェイズB除外）

### Phase C-1: 対象管理と設定基盤

タスク:

1. watchlistスキーマに `evaluation_enabled` と通知粒度設定を追加
2. Firestore repository/API/Webを更新
3. 既存watchlistへのデフォルト移行設計（後方互換）
4. 評価ジョブ実行時刻の全体設定（既定18:00 JST）を追加
5. 月次基礎調査ジョブ実行時刻の全体設定（既定: 毎月1日18:00 JST）を追加

受け入れ基準:

1. 評価対象のON/OFFと通知粒度が銘柄単位で設定できる
2. 既存watchlist機能が回帰しない
3. テスト追加（repository/api/service）

### Phase C-2: 委員会評価エンジン

タスク:

1. 6観点評価ロジックを実装
2. 2軸ラベル（自信×強さ）を算出
3. 売買断言を禁止した出力テンプレートを実装（各観点コメント最大3行）
4. 欠損時は欠損項目を明示する（自信を減衰）
5. 情報源ごとの信頼度スコア算定ロジックを実装

受け入れ基準:

1. 銘柄ごとに6観点と2軸ラベルが出力される
2. 断言文言が含まれない
3. 各観点コメントが3行以内に収まる
4. 単体テストで閾値境界を網羅

### Phase C-3: 通知配信と旧フェイズA置換

タスク:

1. 日次ジョブに委員会評価通知を組み込む
2. watchlist設定に応じて通知粒度を切り替える
3. 旧フェイズA実装を即時置換（退役）する
4. クールダウン、通知ログ、エラー集約を統合する
5. 欠損/更新失敗時に「再取得方法」を通知に含める

受け入れ基準:

1. 旧フェイズAなしで委員会評価通知が運用可能
2. 通知結果JSONに集約される
3. 回帰テストが成功する

### Phase C-4: 運用可視化

タスク:

1. `/notifications/logs` で委員会評価の絞り込みを追加
2. 観点別ヒット件数、確度分布、強さ分布を集計
3. 運用手順へレビュー導線を追加
4. 月次基礎調査の更新失敗一覧と再実行導線（/ops）を追加

受け入れ基準:

1. 1週間分の判定推移をUIまたはログで追える
2. スキップ理由の分析が可能
3. 運用手順書と実装が一致

## 12. Issue分割ガイド（1 Issue = 1目的）

1. `feat: watchlistに委員会評価対象設定を追加`
2. `feat: 委員会6観点評価エンジンを実装`
3. `feat: 2軸ラベル（自信×強さ）を導入`
4. `feat: 通知粒度設定と配信制御を実装`
5. `refactor: 旧フェイズAを委員会評価へ置換`
6. `feat: 委員会評価ログ可視化を追加`
7. `feat: 基礎調査データ3層保存と信頼度スコアを実装`
8. `feat: 月次更新失敗通知と再取得導線を追加`

各Issueに必須記載:

1. 概要
2. 背景/目的
3. スコープ
4. 受け入れ基準（チェックリスト）
5. 依存Issue

## 13. 更新失敗時の運用ルール

1. 基礎調査更新が失敗した場合、前回成功データを継続利用する。
2. 失敗通知には以下を必ず含める。
   - 失敗銘柄
   - 失敗した情報源
   - 最終成功日時
   - 再取得方法（`/ops` で再実行、またはCLIコマンド）
3. 再取得は手動即時実行できるようにする。

## 14. 四季報データ制約時の代替経路

1. 四季報に公式API/許諾がない場合は、手動投入経路を正規サポートする。
2. 手動投入でも `raw/structured/summary` の3層保存を維持する。
3. 手動投入データは `source=manual` を保持し、監査可能にする。
