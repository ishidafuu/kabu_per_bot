# kabu_per_bot 管理画面 使い方手順書

最終更新: 2026-03-08（JST）

この手順書は管理画面の `使い方` ページ（`/guide`）と同等内容です。

## 0. この手順書の対象

1. 日々の通知確認と銘柄管理を行うトレーダー向けです。
2. 「毎日どこを見ればよいか」「通知が来ないときに何を確認するか」を優先して説明します。
3. `/ops`（運用操作）は管理者向け画面です。一般ユーザーが見えない場合は仕様どおりです。

## 1. 最短スタート（3分）

1. 管理画面へアクセス: `https://kabu-per-bot-487501.web.app`
2. メールアドレス/パスワードでログイン（Firebase認証）。
3. `ウォッチリスト` を開き、`新規追加` で銘柄を登録。
4. 最初の推奨設定:
   - 監視方式: `PER`
   - 通知タイミング: `IMMEDIATE`
   - 有効状態: `ON`
5. 一覧に追加されたら初期設定完了。

## 2. 先に押さえる重要ポイント

1. `IMMEDIATE` は「常時リアルタイム監視」ではありません。
2. `IMMEDIATE` は、運用で設定されたジョブ枠で判定されます。
   - 寄り付き帯ジョブ（時間帯/間隔は管理者が `/ops` で設定）
   - 引け帯ジョブ（時間帯/間隔は管理者が `/ops` で設定）
   - 平日18:00の日次ジョブ枠
3. `AT_21` は平日21:05ジョブ枠の判定です。
4. 同一条件通知はクールダウンで抑制されます（既定2時間、管理者が変更可能）。

## 3. 毎日の確認順（実運用）

1. `ダッシュボード`
   - `失敗ジョブ有無` を最初に確認
   - `データ不明件数` が増えていないか確認
2. `通知ログ`
   - 当日通知の有無と通知カテゴリを確認
   - 必要なら ticker で絞り込み（例: `7203:TSE`）
3. `ウォッチリスト`
   - 主要銘柄の `有効状態` / `通知タイミング` / `現在値` を確認
4. `ウォッチリスト` の銘柄詳細
   - `最新テクニカル` で主要指標を確認
   - `技術アラートルール` で銘柄ごとの条件を確認
5. 設定変更した日は `履歴` で操作記録を確認

## 4. ダッシュボードの見方

1. `監視銘柄数`: 登録件数（上限100）
2. `当日通知件数`: 当日JSTの通知件数（0件でも条件未達なら正常）
3. `データ不明件数`: 欠損/取得失敗による `【データ不明】` 通知件数
4. `失敗ジョブ有無`: 日次系ジョブ失敗の有無（`あり` は優先確認）

## 5. ウォッチリスト入力項目

1. `ticker`: `1234:TSE` 形式（例: `7203:TSE`）
2. `会社名`: 一覧表示名
3. `監視方式`: `PER` / `PSR`
4. `通知タイミング`:
   - `IMMEDIATE`: 寄り付き帯/引け帯/18:00枠で判定
   - `AT_21`: 21:05枠で判定
   - `OFF`: 通知停止
5. `優先度`: `HIGH` / `MEDIUM` / `LOW`（一覧・通知ログの絞り込み条件に利用）
6. `常時通知`: 条件未達日も状況通知するか
7. `有効`: 監視対象に含めるか
8. `IR URL`: 改行区切りで登録（候補取得ボタンで候補を追加可能）
9. `AI通知`: 銘柄単位で常時有効（個別ON/OFFは不可）

## 6. 一覧で見るべき列

1. `通知タイミング`: `IMMEDIATE` / `AT_21` / `OFF`
2. `優先度`: `HIGH` / `MEDIUM` / `LOW`
3. `有効状態`: `true` が監視中
4. `現在値`: 当日取得できないと `-`
5. `中央値（1W/3M/1Y）`: 判定基準値
6. `シグナル`: 成立カテゴリと連続日数
7. `通知スキップ理由`: `2時間クールダウン中` / `条件未達（割安シグナルなし）` / `通知タイミングOFF` など
8. `次回決算`: 決算日/時刻（未定の場合あり）
9. `決算まで`: 次回決算までの日数（当日は `当日` 表示）

## 7. 通知ログの使い方

1. `通知カテゴリ` と `条件キー` で通知理由を確認
2. PER/PSR通知の1行目（`優先度` / `推奨` / `判断`）を見て、最初の判断を決める
3. 本文の要約文で `新規` / `継続` / `解除` や決算接近を確認する
4. 最後の `詳細` 行で `PER` と主要期間の乖離率だけ確認する
5. `強通知=true` は優先度の高い通知
6. `データソース` / `取得時刻` で、どの市場データ取得結果を根拠に通知されたかを確認
7. 絞り込みは `ticker` と `優先度` を併用可能
8. 同一条件の連投がないかを時系列で確認

## 7.1 銘柄詳細の見方（技術アラート）

1. `最新テクニカル`
   - `終値-25日線乖離`
   - `出来高倍率`
   - `ATR% (14日)`
   - `20日高値更新` などを確認
2. `技術アラートルール`
   - ルール名、対象指標、演算子、しきい値、有効/無効を確認
   - 指標は `価格系 / トレンド系 / 高値安値系 / 需給系 / 判定フラグ系 / イベント系` などカテゴリ別に選べる
   - `テンプレートから作成` で、25日線回復・出来高急増・20日高値更新などの下書きをすぐ適用できる
   - 必要に応じてルールを追加・編集・無効化する
3. `直近発火履歴`
   - 直近の `技術アラート` 通知本文を確認
   - 条件キーと送信日時が期待どおりかを見る

## 8. よくあるケースと確認順

1. 通知が来ない
   - `ウォッチリスト`: `通知タイミング=OFF` / `有効=false` でないか
   - `ウォッチリスト`: 一覧の `通知スキップ理由` が `クールダウン` / `条件未達` になっていないか
   - `通知ログ`: 直近送信がないか
   - `ダッシュボード`: `失敗ジョブ有無=あり` でないか
2. 通知が少ない
   - 条件未成立なら正常
   - 必要に応じて `常時通知=true` を検討
3. `現在値` が `-`
   - データ欠損の可能性
   - `通知ログ` で `【データ不明】` を確認
   - `【データ不明】` 本文の「次の確認」案内に沿って、必要に応じて `/ops` で対象ジョブ再実行を依頼
4. 銘柄が見つからない
   - 検索条件が残っていないか確認
   - `履歴` で削除/変更有無を確認

## 9. 管理者向け（/ops）の役割

1. 全体設定の管理
   - クールダウン時間
   - IR/SNS通知対象期間（日）
   - `immediate_schedule`（寄り付き帯/引け帯の開始・終了・間隔）
   - Grok SNS取得設定（有効/無効、定時時刻、銘柄ごとの再取得間隔、プロンプト）
2. 手動実行
   - `kabu-immediate-open`
   - `kabu-immediate-close`
   - `kabu-daily`
   - `kabu-daily-at21`
   - `kabu-technical-daily`
   - `kabu-technical-full-refresh`
   - `未計算のみ一括取得` ボタン
     - `latest_technical` が未作成のウォッチリスト銘柄だけをまとめて再同期したいときに使う
   - `kabu-earnings-weekly`
   - `kabu-earnings-tomorrow`
   - Discord疎通テスト
   - Grokクールダウン全解除
   - 補足: `kabu-intelligence` / `kabu-grok` / バックフィル系は定期実行前提のため、通常は手動実行メニューに表示しない
3. 障害時の切り分け起点
   - 実行履歴
   - スキップ理由

## 10. 用語ミニ辞典

1. `PER`: 株価が1株利益の何倍か
2. `PSR`: 株価が1株売上の何倍か
3. `割安判定`: 現在値と1W/3M/1Y中央値の比較結果
4. `クールダウン`: 同条件通知の短時間連投を抑制する仕組み
