このドキュメントは、日常運用で頻出するコマンドを「目的別」にまとめた実行リファレンスです。  
運用時は本書を起点に、必要に応じて詳細手順書へ移動してください。

## 1. 前提と記法

- リポジトリルート: `/Users/ishidafuu/Documents/repository/kabu_per_bot`
- すべてのコマンドは、特記がない限りリポジトリルートで実行
- `<...>` は環境に合わせて置換するプレースホルダ

よく使うプレースホルダ:

- `<FIREBASE_STG_PROJECT_ID>`: Firebase Hosting のステージングプロジェクトID
- `<GCP_PROJECT_ID>`: Cloud Run / Firestore のGCPプロジェクトID
- `<API_STG_SERVICE_NAME>`: ステージングAPIのCloud Runサービス名
- `<ARTIFACT_REGISTRY_IMAGE_URI>`: デプロイ対象コンテナイメージ
- `<DISCORD_WEBHOOK_URL>`: Discord Webhook URL

## 3. 日常運用コマンド（詳細）

### 3.1 同期・状態確認（作業開始時）

```bash

```

解説:

- `--ff-only` によりマージコミットを作らずに更新できる。
- `git status` で、意図しない差分や秘匿情報混入を早期に検知する。

### 3.5 BOT運用コマンド

### 定期実行セットアップ（Cloud Run Jobs + Cloud Scheduler）

```bash
cd /Users/ishidafuu/Documents/repository/kabu_per_bot
bash scripts/setup_scheduler.sh --project-id <GCP_PROJECT_ID>
```

主なオプション:

- `--image-uri <URI>`: 既存イメージを利用（ビルドスキップ）
- `--daily-cron "0 18 * * 1-5"`: 日次ジョブの実行時刻
- `--daily-at21-cron "5 21 * * 1-5"`: 21時向け日次ジョブの実行時刻（平日）
- `--weekly-cron "0 21 * * 6"`: 土曜21時の今週決算通知
- `--tomorrow-cron "0 21 * * *"`: 毎日21時の明日決算通知
- `--backfill-cron "15 21 * * 1-5"`: 平日21:15の増分バックフィル
- `--time-zone Asia/Tokyo`: スケジュールのタイムゾーン

解説:

- 初回作成だけでなく、再実行で設定更新にも対応。
- `DISCORD_WEBHOOK_URL` が環境変数にある場合は Secret へ最新バージョンとして追加する。
- 増分バックフィルJob/Schedulerは `J-Quants APIキー` の Secret（既定: `jquants-api-key`）が利用可能な場合のみ作成する。
- `JQUANTS_API_KEY` が環境変数または `.env` にある場合、`jquants-api-key` Secret に最新バージョンとして追加する。

### 日次ジョブ（PER/PSR・通知）

```bash
cd /Users/ishidafuu/Documents/repository/kabu_per_bot
source .venv/bin/activate
PYTHONPATH=src python scripts/run_daily_job.py --discord-webhook-url <DISCORD_WEBHOOK_URL>
```

補助:

```bash
PYTHONPATH=src python scripts/run_daily_job.py --stdout
PYTHONPATH=src python scripts/run_daily_job.py --jquants-api-key <JQUANTS_API_KEY> --discord-webhook-url <DISCORD_WEBHOOK_URL>
PYTHONPATH=src python scripts/run_daily_job.py --execution-mode at_21 --discord-webhook-url <DISCORD_WEBHOOK_URL>
PYTHONPATH=src python scripts/run_daily_job.py --stdout --no-notification-log
```

Cloud Run Job手動実行（本番環境）:

```bash
gcloud run jobs execute kabu-daily --region=asia-northeast1 --project=<GCP_PROJECT_ID> --wait
gcloud run jobs execute kabu-daily-at21 --region=asia-northeast1 --project=<GCP_PROJECT_ID> --wait
```

管理画面（`/dashboard`）からワンボタン実行する場合の追加設定:

- API環境変数:
  - `API_ADMIN_UIDS=<uid1>,<uid2>`（または Firebase カスタムクレーム `admin=true`）
  - `OPS_GCP_PROJECT_ID=<GCP_PROJECT_ID>`（省略時は `FIRESTORE_PROJECT_ID`）
  - `OPS_GCP_REGION=asia-northeast1`
  - `OPS_DAILY_JOB_NAME=kabu-daily`
  - `OPS_DAILY_AT21_JOB_NAME=kabu-daily-at21`
  - `OPS_EARNINGS_WEEKLY_JOB_NAME=kabu-earnings-weekly`
  - `OPS_EARNINGS_TOMORROW_JOB_NAME=kabu-earnings-tomorrow`
  - `OPS_BACKFILL_JOB_NAME=<任意>`（バックフィル実行ボタンを有効化する場合）
  - `DISCORD_WEBHOOK_URL=<DISCORD_WEBHOOK_URL>`（Discord疎通テストボタン用）
- API実行サービスアカウント権限:
  - `roles/run.developer`（Cloud Run Jobs 実行・履歴取得）
  - `roles/logging.viewer`（スキップ理由集計）

解説:

- 本番相当では Discord URL 指定で実行する。
- `--execution-mode daily` は `IMMEDIATE` 銘柄、`--execution-mode at_21` は `AT_21` 銘柄が対象。
- `--jquants-api-key`（または `JQUANTS_API_KEY`）を設定すると、`J-Quants v2` を先に参照し、欠損時に `四季報online → 株探 → Yahoo!ファイナンス` の順で補完する。
- `--stdout` は文面確認やローカル検証向け。
- `--stdout --no-notification-log` は通知チェック向け（cooldown無効、`notification_log` を読み書きしない）。
- 日次通知は「条件一致時のみ」送信されるため、実行成功でも `sent=0` は正常ケース（`always_notify_enabled=true` の対象がない場合を含む）。

## 4. デプロイ運用コマンド

### 4.1 デプロイ前チェック（必須）

```bash
cd /Users/ishidafuu/Documents/repository/kabu_per_bot
bash scripts/preflight_deploy_check.sh
```

解説:

- 必須CLIの存在、環境変数の不足、`web` の `npm run build` 成否をまとめて確認する。
- 終了コード `0` を確認してからデプロイへ進む。

### 4.2 Webのみ変更時（Firebase Hosting）

```bash
cd /Users/ishidafuu/Documents/repository/kabu_per_bot/web
VITE_USE_MOCK_API=false \
VITE_USE_MOCK_AUTH=false \
VITE_API_BASE_URL="<STG_API_BASE_URL>" \
npm run build

cd /Users/ishidafuu/Documents/repository/kabu_per_bot
firebase use <FIREBASE_STG_PROJECT_ID>
firebase deploy --only hosting
```

解説:

- Web単独変更時の最小手順。
- 先に `npm run build` で破損を検知してから反映する。

### 4.3 API変更時（Cloud Run）

```bash
gcloud config set project <GCP_PROJECT_ID>
gcloud run deploy <API_STG_SERVICE_NAME> \
  --image <ARTIFACT_REGISTRY_IMAGE_URI> \
  --region asia-northeast1 \
  --platform managed \
  --allow-unauthenticated
```

重要ルール:

- `--allow-unauthenticated` は必須（Web Hosting からの直接呼び出しで必要）。
- `--no-allow-unauthenticated` は使わない（`403` の原因になりやすい）。

### 4.4 Cloud Run IAM確認（デプロイ直後）

```bash
gcloud run services get-iam-policy <API_STG_SERVICE_NAME> \
  --region asia-northeast1 \
  --format='table(bindings.role,bindings.members)'
```

確認ポイント:

- `roles/run.invoker` に `allUsers` が含まれていること。

### 4.5 IAM即時復旧（403時）

```bash
gcloud run services add-iam-policy-binding <API_STG_SERVICE_NAME> \
  --region asia-northeast1 \
  --member="allUsers" \
  --role="roles/run.invoker"
```

解説:

- Cloud Run APIをHostingから呼ぶ構成で、`allUsers` invoker欠落を即時復旧する標準コマンド。

### 通知が来ないときの最短切り分け

1. Discord疎通確認: `scripts/send_discord_test_notification.py` が成功するか確認
2. ジョブ実行確認: `gcloud run jobs executions list --job=kabu-daily --region=asia-northeast1 --project=<GCP_PROJECT_ID>`
3. 21時ジョブ確認: `gcloud run jobs executions list --job=kabu-daily-at21 --region=asia-northeast1 --project=<GCP_PROJECT_ID>`
4. スケジュール確認: `gcloud scheduler jobs list --location=asia-northeast1 --project=<GCP_PROJECT_ID>`
5. 設定確認: 対象銘柄の `notify_channel=DISCORD` と `notify_timing`（`IMMEDIATE`/`AT_21`）が実行モードと一致しているか確認
6. 判定確認: 当日条件未達なら通知は送られない（`sent=0` でも正常）。割安未成立日の通知が必要なら `always_notify_enabled=true` を設定
