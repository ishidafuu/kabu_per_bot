## 2.1 実運用フロー（推奨順）

日々の運用は、次の順で実施すると切り分けが早い。

1. `/ops` の全体設定を確認
   - `cooldown_hours`
   - `immediate_schedule`（寄り付き帯/引け帯の開始・終了・間隔）
2. 疎通確認
   - `PYTHONPATH=src python scripts/send_discord_test_notification.py --webhook-url <DISCORD_WEBHOOK_URL>`
3. IMMEDIATE系ジョブを手動実行（必要時）
   - `gcloud run jobs execute kabu-immediate-open --region=asia-northeast1 --project=<GCP_PROJECT_ID> --wait`
   - `gcloud run jobs execute kabu-immediate-close --region=asia-northeast1 --project=<GCP_PROJECT_ID> --wait`
4. AT_21/決算ジョブの状態確認
   - `gcloud run jobs executions list --job=kabu-daily-at21 --region=asia-northeast1 --project=<GCP_PROJECT_ID>`
   - `gcloud run jobs executions list --job=kabu-earnings-weekly --region=asia-northeast1 --project=<GCP_PROJECT_ID>`
5. 通知ログ確認
   - `/notifications/logs` で ticker 絞り込みし、当日送信内容を確認
6. 問題時は「通知が来ないときの最短切り分け」を上から順に実施

### 3.5 BOT運用コマンド

### 定期実行セットアップ（Cloud Run Jobs + Cloud Scheduler）

```bash
cd /Users/ishidafuu/Documents/repository/kabu_per_bot
bash scripts/setup_scheduler.sh --project-id <GCP_PROJECT_ID>
```

主なオプション:

- `--image-uri <URI>`: 既存イメージを利用（ビルドスキップ）
- `--immediate-open-cron "*/1 8-11 * * 1-5"`: 寄り付き帯ジョブの起動間隔
- `--immediate-close-cron "*/1 13-16 * * 1-5"`: 引け帯ジョブの起動間隔
- `--daily-cron "0 18 * * 1-5"`: 日次ジョブの実行時刻
- `--daily-at21-cron "5 21 * * 1-5"`: 21時向け日次ジョブの実行時刻（平日）
- `--weekly-cron "0 21 * * 6"`: 土曜21時の今週決算通知
- `--tomorrow-cron "0 21 * * *"`: 毎日21時の明日決算通知
- `--intelligence-cron "5 21 * * 1-5"`: intelligence(IR)ジョブの実行時刻（既定は平日21:05）
- `--grok-cron "*/1 * * * *"`: Grokジョブの起動間隔（既定は毎分）
- `--backfill-cron "15 21 * * 1-5"`: 平日21:15の増分バックフィル
- `--time-zone Asia/Tokyo`: スケジュールのタイムゾーン

解説:

- 初回作成だけでなく、再実行で設定更新にも対応。
- `DISCORD_WEBHOOK_URL` が環境変数にある場合は Secret へ最新バージョンとして追加する。
- `INTELLIGENCE_IR_SECRET_NAME` / `INTELLIGENCE_SNS_SECRET_NAME` を指定すると、IR/SNS 用Webhook Secretを分離できる（未指定時は共通Secretを利用）。
- `DISCORD_WEBHOOK_URL_INTELLIGENCE_IR` / `DISCORD_WEBHOOK_URL_INTELLIGENCE_SNS`（環境変数または `.env`）がある場合、分離Secretの最新バージョンとして登録する。
- 増分バックフィルJob/Schedulerは `J-Quants APIキー` の Secret（既定: `jquants-api-key`）が利用可能な場合のみ作成する。
- `JQUANTS_API_KEY` が環境変数または `.env` にある場合、`jquants-api-key` Secret に最新バージョンとして追加する。
- `GROK_API_KEY` が環境変数または `.env` にある場合、`grok-api-key` Secret に最新バージョンとして追加し、Grok Job/Schedulerを作成する。
- Grok Secret がない場合は、IR用 `kabu-intelligence` のみ作成する。

管理画面（`/ops`）からワンボタン実行する場合の追加設定:

- API環境変数:
  - `API_ADMIN_UIDS=<uid1>,<uid2>`（または Firebase カスタムクレーム `admin=true`）
  - `OPS_GCP_PROJECT_ID=<GCP_PROJECT_ID>`（省略時は `FIRESTORE_PROJECT_ID`）
  - `OPS_GCP_REGION=asia-northeast1`
  - `OPS_IMMEDIATE_OPEN_JOB_NAME=kabu-immediate-open`
  - `OPS_IMMEDIATE_CLOSE_JOB_NAME=kabu-immediate-close`
  - `OPS_DAILY_JOB_NAME=kabu-daily`
  - `OPS_DAILY_AT21_JOB_NAME=kabu-daily-at21`
  - `OPS_EARNINGS_WEEKLY_JOB_NAME=kabu-earnings-weekly`
  - `OPS_EARNINGS_TOMORROW_JOB_NAME=kabu-earnings-tomorrow`
  - `OPS_BACKFILL_JOB_NAME=<任意>`（バックフィル実行ボタンを有効化する場合）
  - `DISCORD_WEBHOOK_URL=<DISCORD_WEBHOOK_URL>`（Discord疎通テストボタン用）
- API実行サービスアカウント権限:
  - `roles/run.developer`（Cloud Run Jobs 実行・履歴取得）
  - `roles/logging.viewer`（スキップ理由集計）

解説:

- 本番相当では Discord URL 指定で実行する。
- `--execution-mode daily` は `IMMEDIATE` 銘柄、`--execution-mode at_21` は `AT_21` 銘柄が対象。
- `--jquants-api-key`（または `JQUANTS_API_KEY`）を設定すると、`J-Quants v2` を先に参照し、欠損時に `株探 → Yahoo!ファイナンス` の順で補完する。
- `--stdout` は文面確認やローカル検証向け。
- `--stdout --no-notification-log` は通知チェック向け（cooldown無効、`notification_log` を読み書きしない）。
- 日次通知は「条件一致時のみ」送信されるため、実行成功でも `sent=0` は正常ケース（`always_notify_enabled=true` の対象がない場合を含む）。

### 4.3 API変更時（Cloud Run）

```bash

gcloud run deploy <API_STG_SERVICE_NAME> \
  --image <ARTIFACT_REGISTRY_IMAGE_URI> \
  --region asia-northeast1 \
```

重要ルール:

- `--allow-unauthenticated` は必須（Web Hosting からの直接呼び出しで必要）。
- `--no-allow-unauthenticated` は使わない（`403` の原因になりやすい）。

### 4.4 Cloud Run IAM確認（デプロイ直後）

```bash
gcloud run services get-iam-policy <API_STG_SERVICE_NAME> \
  --region asia-northeast1 \
  --format='table(bindings.role,bindings.members)'
```

確認ポイント:

- `roles/run.invoker` に `allUsers` が含まれていること。

### 4.5 IAM即時復旧（403時）

```bash
gcloud run services add-iam-policy-binding <API_STG_SERVICE_NAME> \
  --region asia-northeast1 \
  --member="allUsers" \
  --role="roles/run.invoker"
```

解説:

- Cloud Run APIをHostingから呼ぶ構成で、`allUsers` invoker欠落を即時復旧する標準コマンド。

2. 寄り付き帯ジョブ確認: `gcloud run jobs executions list --job=kabu-immediate-open --region=asia-northeast1 --project=<GCP_PROJECT_ID>`
3. 引け帯ジョブ確認: `gcloud run jobs executions list --job=kabu-immediate-close --region=asia-northeast1 --project=<GCP_PROJECT_ID>`
4. 21時ジョブ確認: `gcloud run jobs executions list --job=kabu-daily-at21 --region=asia-northeast1 --project=<GCP_PROJECT_ID>`
5. スケジュール確認: `gcloud scheduler jobs list --location=asia-northeast1 --project=<GCP_PROJECT_ID>`
6. 設定確認: 対象銘柄の `notify_channel=DISCORD` と `notify_timing`（`IMMEDIATE`/`AT_21`）が実行モードと一致しているか確認
7. 判定確認: 当日条件未達なら通知は送られない（`sent=0` でも正常）。割安未成立日の通知が必要なら `always_notify_enabled=true` を設定
