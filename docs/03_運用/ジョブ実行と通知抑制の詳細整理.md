# ジョブ実行と通知抑制の詳細整理

最終更新: 2026-02-19（JST）

このドキュメントは、Cloud Scheduler/Cloud Run Jobs で起動される各ジョブについて、以下を実装ベースで整理したものです。

- いつ起動されるか
- どの条件で実処理に入るか
- どの条件で通知対象から外れるか
- どの条件で通知がスキップされるか

---

## 1. 先に押さえる全体像

通知までの流れは、次の4段階です。

1. Scheduler起動: Cloud Scheduler が Cloud Run Job を起動する
2. ジョブ内ゲート: 時間帯・定時条件などを満たすか判定する
3. 銘柄フィルタ: `watchlist` の設定で通知対象銘柄を絞る
4. 通知抑制判定: クールダウン等を満たす通知だけ送る

重要ポイント:

- 「ジョブが起動した」ことと「通知が送られる」ことは別です。
- `sent=0` でも、条件未成立なら正常です。
- 「通知対象外（continue）」と「通知スキップ（skipped加算）」は別概念です。

## 3. 設定値の優先順位

ランタイム設定は次の優先順位で解決されます。

1. Firestore `global_settings/runtime`
2. `.env` / 環境変数
3. コード既定値

主な既定値:

- `cooldown_hours=2`
- `window_1w_days=5`
- `window_3m_days=63`
- `window_1y_days=252`
- `intel_notification_max_age_days=30`
- `grok_sns.scheduled_time=21:10`
- `grok_sns.per_ticker_cooldown_hours=24`

## 4. ジョブ別の実処理条件と通知判定

## 4.1 日次系（`run_daily_job.py` / `run_immediate_window_job.py`）

対象ジョブ:

- `kabu-daily`（`--execution-mode daily`）
- `kabu-daily-at21`（`--execution-mode at_21`）
- `kabu-immediate-open`（`--window open`）
- `kabu-immediate-close`（`--window close`）

### 4.1.1 実処理ゲート

`kabu-daily` / `kabu-daily-at21`:

- 常に日次パイプラインを実行します。

`kabu-immediate-open` / `kabu-immediate-close`:

- `immediate_schedule.enabled=false` なら実処理しません。
- 現在時刻が設定時間帯外なら実処理しません。
- 設定間隔分に一致しない分（interval gate）なら実処理しません。
- 上記で実処理しない場合、`processed=0 sent=0 skipped=0 errors=0` で正常終了します。

### 4.1.2 銘柄フィルタ（通知対象外になる条件）

各銘柄は次に該当すると通知判定の対象から外れます。

- `is_active=false`
- `notify_channel=OFF`（またはDiscord以外）
- `notify_timing` がジョブ実行モードと不一致
- `notify_timing=OFF`

モード対応:

- `daily` は `notify_timing=IMMEDIATE` のみ対象
- `at_21` は `notify_timing=AT_21` のみ対象
- `all` は両方対象（手動実行時）

### 4.1.3 通知候補の作られ方

通知候補は主に次の4種類です。

1. 割安通知  
`PER割安` / `超PER割安` / `PSR割安` / `超PSR割安`

2. 状況通知（常時通知ON時）  
`always_notify_enabled=true` かつ割安成立なしの場合に `PER状況` / `PSR状況`

3. 欠損通知  
`【データ不明】`（市場データ取得失敗、必要項目欠損など）

4. 送信なし  
割安非成立かつ常時通知OFFなら通知候補自体を作らない

### 4.1.4 通知スキップ（送信抑制）

通知候補が作られても、以下は送信抑制されます。

- 同一 `ticker + category + condition_key` がクールダウン時間内

補足:

- クールダウンは `notification_log` を参照して判定します。
- 例外として、通常通知から強通知への遷移は即時送信されます。
  - 強通知判定: `under_1y && under_3m && under_1w`

`--stdout --no-notification-log` について:

- 日次系の通知プレビュー専用です。
- クールダウン判定を無効化し、`notification_log` を読み書きしません。

## 4.2 決算系（`run_earnings_job.py`）

対象ジョブ:

- `kabu-earnings-weekly`（来週決算を「今週決算」で通知）
- `kabu-earnings-tomorrow`（翌日決算を「明日決算」で通知）

### 4.2.1 実処理ゲート

- ジョブ自体は常に実行されます。
- `earnings_calendar` から該当日付のエントリを抽出します。

### 4.2.2 銘柄フィルタ

該当日付エントリがあっても、次の銘柄は通知対象外です。

- `watchlist` に存在しない
- `is_active=false`
- `notify_channel=OFF`
- `notify_timing=AT_21` 以外（決算ジョブはAT_21固定で評価）

### 4.2.3 通知スキップ

通知候補（`今週決算` / `明日決算`）は日次と同じクールダウン判定を通ります。

- 同一 `ticker + category + condition_key(EARNINGS:<date>)` がクールダウン時間内ならスキップ

---

## 4.3 IR/SNS/AI系（`run_intelligence_job.py`）

対象ジョブ:

- `kabu-intelligence`（通常は `--intel-source ir_only`）
- `kabu-grok`（通常は `--intel-source grok_only --respect-grok-schedule`）

### 4.3.1 実処理ゲート

IR:

- `ir_only` は毎回実行対象（ソース取得できる限り評価）

Grok:

- `grok_sns.enabled=false` なら実処理しない
- `--respect-grok-schedule` 指定時、実行時刻JSTが `grok_sns.scheduled_time` と一致しない分は実処理しない

### 4.3.2 銘柄フィルタ

日次系と同様に、次の銘柄は対象外です。

- `is_active=false`
- `notify_channel=OFF`
- `notify_timing` が実行モードと不一致
- `notify_timing=OFF`

### 4.3.3 通知候補の作られ方

主な通知カテゴリ:

- `IR更新`
- `SNS注目`
- `AI注目`
- `データ不明`（IR/SNSソース失敗時）

生成条件:

- `IR更新` / `SNS注目`: 新規イベント検知時
- `AI注目`: ウォッチリスト銘柄で常時試行（AI失敗時は速報通知のみ継続）

### 4.3.4 IR/SNS固有のスキップ条件

1. 既読イベント  
`fingerprint` が既に `intel_seen` にあるイベントは再通知しない

2. IR初回実行の既読化  
銘柄ごとのIR初回は「既読化のみ」で通知しない

3. 公開日範囲外  
`published_at` が `intel_notification_max_age_days` より古いイベントは通知しない

4. Grok銘柄別取得抑制  
同一銘柄で `SNS注目` 通知が `per_ticker_cooldown_hours` 内にある場合、Grok取得自体をスキップ

5. 通知クールダウン  
通知候補化されたものは、日次と同じクールダウン判定を受ける

注記:

- `published_at` が取得できないイベントは取りこぼし防止のため対象内として扱います。

---

## 5. 「通知が来ない」理由の分類

運用上は、次の順で切り分けると早いです。

1. 起動前提不足  
Job/Scheduler未作成、Secret不足、Scheduler停止

2. ジョブ内ゲートで停止  
IMMEDIATEの時間帯/間隔不一致、Grok定時不一致

3. 銘柄フィルタで対象外  
`is_active=false`、`notify_channel=OFF`、`notify_timing` 不一致

4. 条件未成立  
割安不成立かつ常時通知OFF、該当決算なし、新規IR/SNSなし

5. 通知抑制でスキップ  
クールダウン中、既読、IR初回既読化、公開日範囲外、Grok銘柄別取得抑制

---

## 6. カウンタの見方（`processed/sent/skipped/errors`）

- `processed`: 銘柄処理に入った件数
- `sent`: 実際に送信した通知件数
- `skipped`: 通知候補は作ったが送信抑制された件数
- `errors`: 銘柄処理中エラー件数

注意:

- 早期に「対象外」で除外された銘柄は `processed` に入らない場合があります。
- `sent=0` は失敗を意味しません。
